# MacMon — спецификация для разработки

> Этот документ — техническое задание для Claude Code. Он описывает menubar-приложение
> мониторинга системы для macOS. Читай его целиком перед началом, следуй разделам по порядку,
> сверяйся с критериями приёмки в конце.

---

## 0. TL;DR для Claude Code

Собери нативное **menubar-приложение для macOS** на **Swift + SwiftUI**, которое в реальном
времени показывает: загрузку CPU, температуру CPU, обороты вентиляторов, заполненность RAM и
скорость сети (download/upload). Данные обновляются автоматически по таймеру (без ручного
рефреша). Есть всплывающее окно (popover) в стиле liquid glass с цветовыми порогами, экран
настроек (автозапуск, интервал опроса, состав индикатора в самом menubar).

**Целевая платформа:** Apple Silicon (M1–M4), macOS 14.0+.
**Без Dock-иконки** (`LSUIElement`), приложение живёт только в строке меню.

Работай итеративно: сначала каркас и легкодоступные метрики (CPU/RAM/сеть), затем SMC-модуль
(температура/вентиляторы) отдельно, потом настройки и полировка UI. Коммить каждый рабочий этап.

---

## 1. Цель и контекст

Пользователю нужен лёгкий «системный пульс» в строке меню Mac:

- **Загрузка CPU** (%)
- **Температура CPU** (°C)
- **Обороты вентиляторов** (RPM, может быть несколько)
- **Заполненность RAM** (использовано / всего, %)
- **Скорость сети** — download и upload (байт/с → человекочитаемо)

Приложение должно быть быстрым, не греть сам процессор частым опросом, выглядеть нативно и
современно, требовать минимум кликов.

## 2. Технические решения (зафиксированы)

| Параметр | Решение |
|----------|---------|
| Язык | Swift (последняя стабильная) |
| UI | SwiftUI |
| Архитектура чипа | **Только Apple Silicon (arm64)** |
| Мин. версия ОС | **macOS 14.0** |
| Menubar API | `MenuBarExtra` (SwiftUI, macOS 13+) со стилем `.window` |
| Dock-иконка | Скрыта: `LSUIElement = YES` в Info.plist |
| Автозапуск | `SMAppService.mainApp` (ServiceManagement, macOS 13+) |
| Доступ к датчикам | IOKit + SMC (см. раздел 6) |
| Хранение настроек | `@AppStorage` / `UserDefaults` |
| Сборка | Xcode-проект (`.xcodeproj`), плюс инструкции в README |
| Зависимости | По минимуму. Чистый Swift + системные фреймворки. SMC — свой код, без внешних либ |

> мягко (показывай «—», не падай).

## 3. Архитектура

Разнеси код по слоям. Один класс = одна ответственность.

```
MacMon/
├── MacMonApp.swift            // @main, MenuBarExtra, LSUIElement
├── Core/
│   ├── MetricsStore.swift     // ObservableObject, @Published-метрики, владелец таймера
│   ├── PollingTimer.swift     // создаёт/пересоздаёт Timer под текущий интервал
│   └── Models.swift           // структуры данных метрик, пороги, enum статусов
├── Sensors/
│   ├── CPUSensor.swift        // загрузка CPU через host_processor_info
│   ├── MemorySensor.swift     // RAM через host_statistics64
│   ├── NetworkSensor.swift    // RX/TX через getifaddrs + дельта по времени
│   └── SMC/
│       ├── SMCService.swift   // высокоуровневый API: temp(), fans()
│       ├── SMCKeys.swift      // известные ключи/сенсоры для Apple Silicon
│       └── SMCBridge.{h,m}    // C/ObjC-мост к IOKit, если нужен
├── UI/
│   ├── MenuBarLabel.swift     // то, что рисуется В СТРОКЕ меню (настраиваемо)
│   ├── PopoverView.swift      // основное окно со всеми метриками (liquid glass)
│   ├── MetricRow.swift        // строка: иконка + лейбл + бар со скруглением + значение
│   ├── NetworkFooter.swift    // блок down/up
│   ├── SettingsView.swift     // экран настроек
│   └── Theme.swift            // цвета, пороги, материалы, константы стиля
├── Services/
│   └── LaunchAtLogin.swift    // обёртка над SMAppService
├── Resources/
│   └── Assets.xcassets        // иконка menubar (SF Symbol или template image)
└── Info.plist
```

**Поток данных:** `PollingTimer` тикает → сенсоры читают системные значения → `MetricsStore`
обновляет `@Published`-свойства → SwiftUI автоматически перерисовывает `MenuBarLabel` и
`PopoverView`. Никаких ручных вызовов рефреша из UI (кнопка refresh — лишь форсирует один тик).

## 4. Метрики: что читать и как

### 4.1 Загрузка CPU
- API: `host_processor_info(mach_host_self(), PROCESSOR_CPU_LOAD_INFO, ...)`.
- Считай **дельту** тиков (user+system+nice) между двумя опросами, дели на общую дельту.
- Отдавай агрегированный % по всем ядрам (0–100). Опционально — per-core, но не в MVP.

### 4.2 Заполненность RAM
- API: `host_statistics64(..., HOST_VM_INFO64, ...)` → `vm_statistics64_data_t`.
- Использовано ≈ `(active + wired + compressed) * page_size`. Всего — через
  `ProcessInfo.processInfo.physicalMemory`.
- Отдавай: использовано (GB), всего (GB), процент.

### 4.3 Скорость сети (RX/TX)
- API: `getifaddrs()` → суммируй `if_data.ifi_ibytes` / `ifi_obytes` по активным интерфейсам
  (исключи `lo0`).
- Скорость = (текущие байты − предыдущие байты) / прошедшее время.
- Форматируй: B/s → KB/s → MB/s (человекочитаемо, 1 знак после запятой).
- Веди previous-снимок внутри `NetworkSensor`.

### 4.4 Температура CPU и вентиляторы — см. раздел 6 (SMC).

## 5. Цветовые пороги

Каждая метрика имеет три зоны статуса. Реализуй как единый механизм в `Theme.swift` / `Models.swift`.

```swift
enum MetricStatus { case normal, warm, hot }  // зелёный, янтарный, красный
```

| Метрика | normal | warm | hot |
|---------|--------|------|-----|
| Температура CPU | < 70 °C | 70–85 °C | > 85 °C |
| Загрузка CPU | < 60 % | 60–85 % | > 85 % |
| RAM | < 70 % | 70–90 % | > 90 % |
| Вентиляторы | < 70 % от макс | 70–90 % | > 90 % |

- Цвета: normal `#6FE0B8` (teal/green), warm `#FFB84D` (amber), hot `#FF6B5D` (red).
- Порог температуры должен быть **настраиваемым** (граница warm/hot регулируется в настройках).
- При статусе `hot` — заливка бара с лёгким свечением (см. раздел 7).

## 6. SMC-модуль (температура и вентиляторы) — самая сложная часть

> Выделено отдельно намеренно. Это главный риск проекта. Сделай так, чтобы остальное
> приложение работало даже если SMC ничего не вернул.

### Требования к реализации
1. Целевая платформа — **Apple Silicon**, поэтому используй современный путь чтения сенсоров.
   Рекомендуемый подход: `IOHIDEventSystemClient` с фильтром по `kHIDPage_AppleVendor`
   (температурные и тахо-сенсоры). Альтернатива/дополнение — обращение к `AppleSMC`
   через IOKit (`IOServiceGetMatchingService` → `IOConnectCallStructMethod`) для ключей
   вентиляторов.
2. Не хардкодь Intel-ключи как основной путь. Они на M-чипах не дают валидных значений.
3. Сделай **discovery**: перечисли доступные сенсоры, отфильтруй температурные (CPU-кластеры)
   и тахо (вентиляторы). На некоторых Mac (особенно ноутбуках без активного охлаждения,
   например MacBook Air) **вентиляторов нет** — обрабатывай это как валидное «нет данных».
4. Верни из `SMCService`:
   - `cpuTemperature: Double?` (агрегат/максимум по доступным CPU-сенсорам)
   - `fans: [FanReading]` где `FanReading { current: Int, min: Int, max: Int }`
5. **Деградация:** если значение недоступно — `nil`, в UI показывается «—», приложение не падает.
6. Не требуй `sudo`. Если выбранный путь требует повышенных прав — выбери другой
   (powermetrics с sudo — НЕ подходит).

### Безопасность реализации
- Оберни все C-вызовы аккуратно, проверяй коды возврата IOKit.
- Освобождай ресурсы (`IOObjectRelease`, `IOServiceClose`).
- Опрос SMC дороже остальных метрик — учитывай это: можно опрашивать температуру/вентиляторы
  реже, чем CPU/сеть, если пользователь выбрал интервал 1с (например, температуру — раз в 2с).

### Если discovery не дал результата
Залогируй доступные сенсоры в консоль (для отладки) и оставь TODO с инструкцией, какие ключи
проверить вручную на конкретной модели чипа. Не выдумывай значения.

## 7. UI / дизайн

Стиль — **liquid glass**: полупрозрачное матовое стекло с блюром, НЕ полностью прозрачное.

### 7.1 Материал
- Используй системный материал, а не CSS-имитацию: `NSVisualEffectView` с `.material = .popover`
  или `.hudWindow`, либо SwiftUI `.background(.ultraThinMaterial)`.
- Тонкая светлая граница сверху (внутренний хайлайт), мягкая тень окна.
- Автоматическая адаптация к светлой/тёмной теме macOS — обязательно.

### 7.2 Popover (главное окно)
Ширина ~340pt. Сверху вниз:
1. Заголовок: иконка-пульс + «MacMon», справа — кнопки refresh и settings (шестерёнка).
2. Строки метрик (`MetricRow`), каждая: SF Symbol слева, лейбл, значение справа (моноширинный),
   под ними — прогресс-бар.
3. Блок сети внизу (две колонки: Down / Up со стрелками).

### 7.3 MetricRow — прогресс-бары
- Высота бара ~7pt, трек полупрозрачный.
- **Заливка скруглена на правом конце** (`border-radius: 0 99px 99px 0` по смыслу — в SwiftUI
  это `Capsule()`/`RoundedRectangle` с обрезкой левого края, либо clip заполненной части
  капсулой). Левый край прямой, правый — закруглён.
- Цвет заливки = цвет статуса (раздел 5). При `hot` — добавить мягкое свечение (`shadow`/glow).
- Значение в строке тоже окрашивается в цвет статуса при warm/hot.

### 7.4 MenuBar label — настраиваемо пользователем
То, что видно в самой строке меню (рядом с иконкой), настраивается:
- Режимы: «только иконка», «иконка + 1–2 значения (напр. CPU% / temp°)»,
  выбор какие именно метрики показывать.
- Компактный текст, моноширинный, чтобы не «прыгал» по ширине.
- Иконка — template image (SF Symbol), чтобы корректно тонироваться в тёмной/светлой строке меню.

### 7.5 Палитра (для согласованности)
- Фон-стекло: системный материал.
- Акценты: CPU `#5DB0FF`, температура — по статусу, RAM — по статусу, сеть down `#7FC0FF`,
  up `#FF9D7A`.
- Текст на стекле — высокий контраст, адаптивный к теме.

## 8. Настройки (`SettingsView`)

Окно/панель настроек, доступная из popover. Содержит:

1. **Запуск при входе** — тумблер.
   - `SMAppService.mainApp.register()` / `.unregister()`.
   - Отрази реальный текущий статус (`SMAppService.mainApp.status`).
   - Обработай ошибки регистрации (покажи сообщение, не падай).

2. **Интервал опроса** — сегментированный выбор: 1с / 2с / 5с (дефолт **2с**).
   - При смене — пересоздать таймер с новым интервалом.
   - Сохранять в `UserDefaults`.

3. **Порог температуры** — настраиваемая граница warm/hot (слайдер, напр. 75–95 °C).

4. **Состав menubar-индикатора** — что показывать рядом с иконкой (см. 7.4).

Все настройки персистентны (`@AppStorage`).

## 9. Динамика данных (явно)

- Данные **обновляются автоматически** по таймеру — пользователю НЕ нужно рефрешить.
- `MetricsStore` — `ObservableObject` с `@Published` полями; SwiftUI перерисовывает сам.
- Кнопка refresh в popover — опциональна, только форсирует один внеочередной тик.
- Таймер останавливается, когда это разумно (можно не усложнять в MVP — пусть тикает всегда,
  опрос дешёвый, кроме SMC).

## 11. Порядок работы (этапы для Claude Code)

Делай по шагам, проверяй сборку после каждого, коммить.

1. **Каркас:** Xcode-проект, `MenuBarExtra`, пустой popover, `LSUIElement`. Запускается, видна иконка.
2. **CPU + RAM:** `CPUSensor`, `MemorySensor`, `MetricsStore`, таймер. Значения видны и обновляются.
3. **Сеть:** `NetworkSensor` с дельтой RX/TX, форматирование скорости.
4. **UI v1:** `MetricRow` с барами (скругление справа), liquid-glass материал, блок сети.
5. **Пороги и цвета:** статусы метрик, окраска баров/значений, свечение при hot.
6. **SMC-модуль (отдельно):** discovery сенсоров, температура, вентиляторы, мягкая деградация.
   Это самый рискованный шаг — изолируй его, не ломая остальное.
7. **Настройки:** автозапуск, интервал опроса, порог температуры, состав menubar-индикатора.
8. **MenuBar label настраиваемый:** режимы отображения рядом с иконкой.
9. **Полировка:** тёмная/светлая тема, граничные случаи (нет вентиляторов, SMC вернул nil),
   производительность опроса, README.

## 13. Явные ограничения и риски (чтобы не тратить время)

- **SMC на Apple Silicon недокументирован.** Часть температурных сенсоров может быть недоступна
  без приватных API. Цель — получить хотя бы один валидный CPU-температурный сенсор и обороты
  вентиляторов там, где они физически есть. Где нет — «—».
- **MacBook Air** и некоторые модели — без активных вентиляторов. Это норма, не баг.
- **powermetrics** даёт богатые данные, но требует sudo — НЕ использовать.
- **Per-core CPU, GPU-температура, энергопотребление по компонентам** — вне MVP (можно оставить
  TODO-хуки в архитектуре, но не реализовывать сейчас).
- Не тяни тяжёлые внешние зависимости. SMC-логику пиши сам.

## 14. Заметки по качеству кода

- Изолируй системные/C-вызовы за чистыми Swift-интерфейсами (протоколы), чтобы UI не знал про IOKit.
- Все сенсоры — за протоколом `Sensor` с методом чтения, чтобы их можно было мокать.
- Никаких утечек ресурсов IOKit/Mach.
- Опрос — вне главного потока, обновление UI — на главном (`@MainActor`).
- Коммить осмысленными порциями по этапам раздела 11.
